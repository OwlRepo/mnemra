## Production-RAG roadmap (order locked: scale → measure → LangGraph)

For high-volume BPO (many agents, tons of prompts/user), cost + latency dominate. Build in this order:
- **Stage 1a — Answer caching (THIS handoff):** exact (Redis) + semantic (pgvector) cache with KB-version invalidation. Repetitive support questions hit cache → $0 + instant.
- **Stage 1b — Per-tenant limits (next handoff):** per-user + per-workspace chat rate limit + monthly token/cost budget.
- **Stage 2 — RAGAS eval harness:** score faithfulness/precision/recall from LangSmith traces before optimizing.
- **Stage 3 — Conditional LangGraph:** straight-line RAG by default; escalate to grade→rewrite→retry→self-grade ONLY when retrieval score < threshold (free gate); add "I don't know / escalate to human" routing. Keeps the expensive path rare at volume.

Design rule for the whole stage: the query embedding is computed ONCE per question and reused for semantic-cache lookup AND retrieval (no double embed).

## Task Summary (Stage 1a — Answer caching)

Add a two-tier answer cache to the chat path so repeated/paraphrased support questions skip retrieval + LLM generation: exact-match via Redis (instant, free) then semantic-match via pgvector (catches paraphrases). Invalidate per workspace when its knowledge base changes. Cache lives in `apps/api` (it has workspace context + Redis + db); `packages/ai` retrieval stays unchanged.

- `setExact(workspaceId, message, answer, sources): Promise<void>` → `SET key {answer,sources} EX ${CHAT_CACHE_TTL_SECONDS}` (default 3600).
- `getSemantic(workspaceId, embedding): Promise<{ answer; sources } | null>` → cosine search `chat_cache` within `(workspaceId, version)`; if top `score >= SEMANTIC_CACHE_THRESHOLD` (default 0.95) return it (raw SQL mirroring `similaritySearch`’s `<=>` operator).
- `saveSemantic(workspaceId, version, message, embedding, answer, sources): Promise<void>` → insert into `chat_cache`.

## Risk Register Notes

Deep, hot path. Top risks + mitigations: (1) Cross-tenant leakage — cache keys + the pgvector query MUST include `workspaceId`; tested explicitly. (2) Stale answers after a KB update — per-workspace `version` bumped on document done/delete invalidates instantly; plus a TTL backstop. (3) A wrong semantic hit (paraphrase that isn't equivalent) — threshold defaults high (0.95); make it env-tunable; conservative. (4) Redis unavailable — cache get/set must fail-soft (treat as miss, log) so chat still works without Redis. (5) Don't cache the "I don't have enough information" fallback as if it were a real answer if the KB is just empty — acceptable to cache, invalidated on first ingest via version bump.

## Guardrails

- Cache keys + semantic query MUST be workspace-scoped; never global.
- Fail-soft on Redis/cache errors (degrade to a normal LLM answer; never 500 the chat because the cache is down).
- Do not change `packages/ai` retrieval/answer logic.
- Persist conversation history identically for hits and misses.
- Compute the query embedding once; reuse for semantic lookup (and pass through if you later refactor retrieval to accept it — optional).
- Do not modify unlisted files.
